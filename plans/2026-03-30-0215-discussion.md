# Architecture discussion (2026-03-30)

Notes from design conversations so we can pick up where we left off.

## Engine as the vim brain

All vim semantics (modes, key interpretation, cursor bounds, future buffer operations) live in Go. Renderers (web DOM, future TUI) stay dumb: they subscribe to events and paint state they are told about.

## Granular events over generic `OnAction`

Use small methods on the `EventListener` interface (e.g. `OnModeChanged`, `OnCursorMoved`). The Go compiler enforces that every listener implements them. Avoid a single generic `OnAction` that moves the switch statement to JS and loses type safety on the Go side.

## Engine owns state (Model–View, not HATEOAS)

- **Mode** must live in the engine: the same key means different things in normal vs insert (e.g. `j` vs typing `j`).
- **Cursor position** (grid cell) is authoritative in the engine. The DOM showing which cell has `data-cell-variant="normal"` is a *derived view* of that state, not a second source of truth.
- Multiple UIs (web, TUI) are views over the same model—same pattern as a game engine or MVC.

This is not REST/HATEOAS: the client is not discovering next actions from hypermedia alone; the engine is a modal interpreter with owned state.

## `InsertPosition` as a semantic hint

The engine translates keys into intent: `i` → insert caret **before** current cell text, `a` → **after**. Renderers receive `insertPosition: "before" | "after"` (or empty when irrelevant) and map that to `setSelectionRange` or TUI equivalents—no vim key knowledge in the renderer.

## Future: Buffer and SQLite

- Load query results into an in-memory **buffer** (`[][]string` or similar).
- Vim-style edits mutate the buffer; grid dimensions come from the buffer.
- **`:w`** (or equivalent) flushes dirty cells back to SQLite.

## Structural changes: coarse `OnBufferChanged` (future)

For row insert/delete, paste, etc., a single event carrying the full buffer (plus cursor) lets renderers replace the table body in one shot. Lighter events (`OnCursorMoved`, `OnModeChanged`) stay for the common case where the grid shape does not change.

## Bounds checking in the engine

`hjkl` updates are clamped to `[0, cols-1] × [0, rows-1]`. If a move would leave the grid, the engine does **not** update the cursor and does **not** emit `OnCursorMoved`. The renderer never has to handle out-of-bounds coordinates.

Dimensions are passed into `New(cols, rows)` to match the rendered table until a `Buffer` type owns dimensions.
